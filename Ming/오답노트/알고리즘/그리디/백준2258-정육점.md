#### 1. 정렬 기준의 '디테일' 설정 (복합 조건)
- **교훈:** 단순히 하나의 기준(가격)만으로 정렬하는 것이 아니라, **'동일한 값이 나왔을 때의 보조 기준'**이 승패를 가른다.
- **적용:** * **가격:** 오름차순 (싼 것부터 확인해야 하므로)
    - **무게:** **내림차순** (같은 가격이라면 무게가 큰 것을 먼저 선택해야, 최소 비용으로 최대 효율을 내는 지점을 빨리 찾을 수 있음)
- **기억할 것:** `(o1, o2) -> o1[1] == o2[1] ? o2[0] - o1[0] : o1[1] - o2[1]`
#### 2. 문제 조건의 '예외적 비용 계산' (누적 vs 리셋)
- **교훈:** "덤"으로 주는 조건이 있을 때는 **'비용이 바뀌는 시점'**을 정확히 파악해야 한다.
- **적용:**
    - **이전보다 비싼 고기를 살 때:** 현재 고기 가격만 지불 (이전 고기들은 공짜이므로 비용 리셋).
    - **이전과 같은 가격의 고기를 살 때:** 현재 고기 가격을 **추가로 지불** (누적 합산).
- **실수 포인트:** "같은 가격일 때 넘어가기(continue)"는 문제 조건을 잘못 해석한 것임. 같은 가격일 때 오히려 비용이 계속 늘어난다는 점이 이 문제의 핵심 함정.
#### 3. '조기 종료(Early Exit)'의 위험성
- **교훈:** 그리디 문제에서 "조건을 만족하는 첫 번째 순간이 최적해(Optimal Solution)가 아닐 수 있다"는 점을 항상 의심하라.
- **적용:** 무게 M을 넘긴 시점이 10원(1원짜리 10개 합산)이었는데, 그다음 고기가 5원(5원짜리 1개)일 수도 있음.
- **해결책:** 무게를 채운 순간 `return` 하지 말고, **전체 반복문을 돌며 `Math.min()`으로 최솟값을 계속 업데이트**해야 함.
#### 4. 데이터 범위와 자료형 (Overflow 방지)
- **교훈:** 입력값이 231−1(약 21억) 근처라면, 누적 합산은 무조건 `long`을 고려하라.
- **적용:** 무게 M과 고기들의 무게 합, 그리고 최종 비용은 `int` 범위를 초과할 가능성이 매우 큼.
- **습관:** "N이 크거나 누적합이 필요하면 일단 `long`으로 시작하자."
