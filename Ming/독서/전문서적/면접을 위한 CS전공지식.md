1. 상위 모듈은 하위 모듈의 어떤 것도 가져오지 말아야한다.

3. 추상화는 세부사항에 의존하지 말아야 한다.

#### 3. 팩토리 패턴
1. **객체 생성 부분을 떼어내 추상화한 패턴**으로 상속 관계에 있는 두 클래스 간에 **상위 클래스가 중요한 뼈대를 결정**하고 **하위 클래스가 구체적인 객체 생성에 대한 내용을 결정**하는 패턴.

#### 4. 전략 패턴
1. 객체의 행위를 바꾸고 싶은 경우 **직접 수정하지 않고** 전략이라고 부르는 **캡슐화한 알고리즘**을 **컨텍스트 안에서 바꿔**주면서 상호 교체가 가능하게 만드는 패턴.

#### 5. 옵저버 패턴
1. 주체가 어떤 객체의 상태변화를 관찰하다가 변화가 있을 때 메서드 등을 통해 옵저버 목록에 있는 옵저버들에게 변화를 알려주는 디자인 패턴이다.

#### 6. 프록시 패턴
1. 프록시 패턴은 대상 객체에 접근하기 전 그 접근에 대한 흐름을 가로채어 해당 접근을 필터링하거나 수정하는 역할을 함. 
	1. 캐싱, 검증, 로깅에 사용
2. 데코레이터 패턴과 유사하지만 의도에서 차이가 있음. 데코레이터 패턴은 기능 추가에 중점을 두고 있다면 프록시 패턴은 필터에 중점을 두고 있음.
#### 7. 프록시 서버
- Forward Proxy 서버는 클라이언트 앞에 놓여져 있는 반면,
- Reverse Proxy 서버는 웹서버/WAS 앞에 놓여 있다는 차이점이 있다.

#### 8. 용어정리
1. CDN : 각 사용자가 인터넷에 접속하는 곳과 가까운 곳에서 컨텐츠를 캐싱 또는 배포하는 서버 네트워크를 말한다.
2. CORS(CROSS-ORIGIN-RESOURCE-SHARING) : 서버가 웹 브라우저에서 리소스를 로드할 때 다른 오리진을 통해 로드하지 못하게 하는 HTTP 헤더 기반 메커니즘

## 프로그래밍 패러다임
#### 선언형과 함수형 프로그래밍
1. 선언형 프로그래밍이란 무엇을 풀어내는가에 집중하는패러다임
2. 함수형 프로그래밍은 작은 순수 함수들을 블록처럼 쌓아 로직을 구현하고 [[고차 함수]]를 통해 재사용성을 높인 프로그래밍 패러다임이다.

#### 객체지향 프로그래밍
- 객체지향 프로그래밍은 추상화, 캡슐화, 상속성, 다형성이라는 특징이 있다.
1. 추상화 : 복잡한 시스템으로 부터 핵심적인 개념 또는 기능을 가추려내는 것.
2. 캡슐화 : 객체의 속성과 메서드를 하나로 묶고 일부를 외부에 감추어 은닉하는 것.
3. 상속성 : 상위 클래스의 특성을 하위 클래스가 이어 받아 재사용하거나 확장하는 것.
4. 다형성 : 하나의 메서드나 클래스가 다양한 방법으로 동작하는 것. 대표적으로 오버로딩과 오버라이등이 있음.
	1. 오버로딩 : 같은 이름의 메서드를 여러 개 두는 것
	2. 오버라이딩 : 상위 클래스로부터 상속받은 메서드를 하위클래스에서 재정의 하는 것.

#### 설계 원칙(SOLID)
1. 단일 책임 원칙(SRP) : 모든 클래스는 각각 하나의 책임만 가져야 하는 원칙
2. 개방-폐쇄 원칙(OCP) : 유지 보수 사항이 생긴다면 코드를 쉽게 확장할 수 있도록 하고 수정할 때는 닫혀 있어야 하는 원칙
3. 리스코프치환 원칙(LSP) : 서브 클래스는 언제나 자신의 부모 클래스를 대체할 수 있어야 한다.
4. 인터페이스 분리 원칙(ISP) : 하나의 일반적인 인터페이스보다 구체적인 여러 개의 인터페이스를 만들어야하는 원칙
5. 의존 역전 원칙(DIP) : 추상화에만 의존하는 설계 원칙

## TCP/IP
#### TCP/IP4 계층
1. 어플리케이션 계층 : 응용 프로그램이 사용되는 프로토콜 계층이며 실질적으로 서비스가 제공되는 계층
2. 전송 계층 : 송신자와 수신자를 연결하는 통신 서비스를 제공하며 어플리케이션과 인터넷 계층 사이 중계자 역할을 한다. 
	1. TCP : 패킷 사이 순서를 보장하고 연결지향 프로토콜을 사용해서 연결하는 가상회선 패킷 교환 방식을 사용
	2. UDP : 순서를 보장하지 않고 수신 여부를 확인하지 않는  데이터그램 패킷 교환 방식 사용
3. 인터넷 계층
4. 링크 계층

#### IP주소
1. ARP : IP주소를 MAC주소로 변경
2. RARP : MAC주소를 IP주소로 변경
3. 게이트웨이 : 서로다른 통신망, 프로토콜을 사용하는 네트워크 간의 통신을 가능하게 하는 관문


## HTTP
#### HTTP1.0
1. 한 연결당 하나의 요청만 처리하도록 설계되어서 RTT증가를 불러옴
2. 대안
	1. 이미지 스플리팅
	2. 코드압축
	3. Base64인코딩
#### HTTP1.1
- keep-alive추가됨
- 무거운 헤더 구조로 무거웠으며 파일이 순서대로 받아와졌기에 앞의 파일에 지연이 있으면 늦게 받아지는 문제가 있었다. (HOL Blocking)

#### HTTP2
-  HTTP1.1의 지연시간을 줄이고 응답시간을 더 빠르게 할 수 있으며 멀티플렉싱, 헤더압출, 서버푸시, 요청의 우선순위 처리를 지원하는 프로토콜이다
1. 멀티플렉싱 : 하나의 연결 내 여러 스트림을 송수신 함으로써 단일 연결을 사용하여 병렬로 여러 요청을 받을 수 있고 응답할 수 있다.
2. 서버 푸시 : html을 읽으면서 그 안에 들어있던 CSS파일을 서버에서 푸시하여 클라이언트에 먼저 줄 수 있다.

## 운영체제
### 캐시
##### 1. 지역성 원리
-  캐시를 직접 설정한다면 자주 사용하는 데이터를 기반으로 설정 해야된다. 자주 사용하는 데이터를 판단하는 기준은 지역성 원리를 이용한다
1. 시간 지역성 : 최근 사용한 데이터에 다시 접근하려는 특성
2. 공간 지역성 : 최근 접근한 데이터를 이루고 있는 공간이나 그 가까운 공간에 접근하는 특성

##### 2. 웹 브라우저의 캐시
1. 쿠키 
	1. 만료기한이 있는 키-값 저장소
	2. 4kb까지 데이터를 저장할 수 있음
	3. 클라이언트, 서버 둘 다 수정 가능
2. 로컬 스토리지
	1. 만료기한이 없는 키-값 저장소
	2. 5mb까지 저장 가능
	3. 브라우저를 닫아도 유지
	4. HTML5를 지원하는 브라우저만 사용
	5. 클라이언트에서만 수정 가능
3. 세션 스토리지
	1. 만료기한이 없는 키-값 저장소
	2. 탭 단위 세션 스토리지 생성, 탭 닫을 때 해당 데이터도 삭제
	3. 5MB까지 저장 가능
	4. HTML5를 지원하는 브라우저만 사용
	5. 클라이언트에서만 수정 가능

### 메모리 관리
#### 1. 가상 메모리
- 컴퓨터가 실제 이용 가능한 메모리 자원을 추상화하여 이를 사용하는 사용자에게 매우 큰 메모리로 보이게 만드는 것
1. 스와핑 : 페이지 폴트가 발생했을 때, 디스크에서 해당 [[페이지]]를 **물리 메모리로 불러오는 과정**
2. 페이지 폴트 : 프로세스의 주소 공간에는 존재하지만 RAM에는 없는 데이터에 접근했을 경우 발생

##### 2. 스레싱
- 메모리의 페이지 폴트율이 높은 것을 의미함
- 스레싱은 메모리에 너무 많은 프로세스가 동시에 올라가면 스와핑이 많이 일어나서 발생하는 것.
- 해결법
	1. 작업세트
	2. PFF
##### 3. 메모리 할당
1. 연속 할당 : 메모리에 연속적으로 공간을 할당하는 것
	1. 고정 분할 방식 : 내부 단편화
	2. 가변 분할 방식 : 외부 단편화(최초적합, 최적적합, 최악적합)
2. 불연속 할당
	1. 페이징 : 동일한 크기의 페이지 단위로 나누어 메모리의 서로 다른 위치에 프로세스를 할당하는 것 (주소변환이 복잡해짐, 홀의 크기가 균일하지 않은 문제 해결)
	2. 세그멘테이션 : 페이지 단위가 아닌 의미 단위인 세그먼트로 나누는 방식 (공유와 보안 측면에 장점이 있지만 홀 크기가 균일하지 않음)

##### 4. 페이지 교체 알고리즘
1. FIFO
2. LRU
3. LFU
