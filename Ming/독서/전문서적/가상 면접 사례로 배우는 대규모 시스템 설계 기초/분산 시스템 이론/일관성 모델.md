## 모델
#### 강한 일관성
- 모든 읽기 연산은 가장 최근에 갱신된 결과를 반환한다.
- 즉, 클라이언트는 절대로 낡은 데이터를 보지 못한다.
#### 약한 일관성
- 읽기 연산은 가장 최근에 갱신된 결과를 반환하지 못할 수 있다.
#### 최종 일관성
- 약한 일관성의 한 형태로, 갱신 결과가 결국에는 모든 사본에 반영되는 모델이다.
- 최종 일관성의 경우 쓰기 연산이 병렬적으로 발생하면 시스템에 저장된 값의 일관성이 깨질 수 있는데 해당 문제는 **클라이언트측에서 데이터의 버전 정보를 활용해서 일관성을 유지**해야한다.

## 충돌 해결 전략: 데이터 버저닝
#### 버저닝
- 데이터를 변경할 때마다 해당 데이터의 새로운 버전을 만드는 것
- 각 버전의 데이터는 변경 불가능하다.
#### 벡터 시계
- (서버, 버전)의 순서쌍을 데이터에 매단 것이다.
- 어떤 버전이 선행 버전인지, 후행 버전인지, 아니면 다른 버전과 충돌이 있는지 판별하는데 쓰인다.
#### 충돌 상황
1. **상황:** 사용자 정보를 담고 있는 데이터가 있습니다.
2. **서버 A의 작업:** 사용자가 이름을 수정해서 서버 A에 저장되었습니다. 이를 v1이라고 합시다.
3. **서버 B의 작업:** 동시에(혹은 서버 A의 변경 사항이 서버 B에 전달되기 전에) 사용자가 전화번호를 수정해서 서버 B에 저장되었습니다. 이를 v2이라고 합시다.
4. **문제 발생:** 나중에 시스템이 v1과 v2를 합치려고 보니, "이름이 바뀐 게 먼저인가? 아니면 전화번호가 바뀐 게 먼저인가?"를 알 수 없게 됩니다.
#### 무엇이 충돌인가?
- **인과 관계가 있는 경우:** v1을 수정한 뒤에 그 데이터를 바탕으로 v2를 만들었다면, v2가 최신임이 명확합니다.
- **충돌(v1,v2):** 두 버전이 서로의 존재를 모르는 상태에서 각자 생성되었다면, 어느 한쪽이 다른 쪽의 '과거'라고 말할 수 없습니다. 이것이 바로 **해소하기 어려운 충돌**입니다.
