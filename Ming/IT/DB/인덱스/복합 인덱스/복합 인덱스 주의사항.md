## 칼럼의 순서 중요성
복합 인덱스의 동작 원리는 우리가 실생활에서 사용하는 '전화번호부'나 '국어사전'과 같다.
- **전화번호부**: '성(Last Name)'으로 먼저 정렬된 후, 같은 성 안에서 '이름(First Name)'으로 다시 정렬된다.
- **국어사전**: '첫 번째 글자'로 먼저 정렬된 후, 같은 첫 글자로 시작하는 단어들끼리 '두 번째 글자'로 다시 정렬된다.
![[Pasted image 20251012155028.png]]
- `category`로 검색: 매우 효율적이다.
- `category`와 `price`로 검색: 매우 효율 적이다.
- `price`만으로 검색: 매우 비효율 적이다.
	- `category`는 1차 정렬되었고 `price`는 2차 정렬되어서  인덱스를 제대롷 활용할 수 없다.

## 복합 인덱스 정렬
```sql
explain 
	select * 
	from items 
	where category = '전자기기' 
	and price >= 100000 
	order by price;
```
- `ORDER BY`를 사용할 때 복합 인덱스의 순서대로 정렬하면 추가적인 정렬(`filesort`)를 피할 수 있다.

## 등호 조건은 앞으로, 범위 조건은 뒤로
>선행 칼럼에 범위 조건(`>,<, BETWEEN, LIKE %`)이 사용되면 그 뒤에 오는 칼럼은 인덱스를 제대로 활용할 수 없다.
#### 복합 인덱스 실행 실패 과정
```sql
EXPLAIN 
	SELECT * 
	FROM items 
	WHERE category >= '패션' 
	AND price = 20000;
```

![[Pasted image 20251012163612.png]]
1. `idx_items_category_price` 인덱스를 사용해 `category` 가 '패션'인 위치를 찾는다. 
	- (`>=` 조건의 시작점)
2. 거기서부터 인덱스의 끝까지 **모든 데이터를 스캔**한다. 
	- ('패션', '헬스/뷰티' 카테고리에 해당하는 모든 데이터)
3. 스캔하는 각 레코드마다 `price = 20000` 조건을 만족하는지 **하나하나 검사**한다. 
	- (인덱스 사용이 아니라 직접 필터링 한다.)

#### 선행 칼럼을 범위조건으로 사용했을 때 문제점
데이터베이스에서 `category >= '패션'`조건에 따라 인덱스에서 '패션'과 '헬스/뷰티'섹션을 찾았다.
하지만 `price = 20000`을 처리할 때 문제가 발생한다.
- '패션'섹션은 `price`순으로 정렬되어 있다.
- '헬스/뷰티' 섹션도 `price`순으로 정렬되어있다.
'패션'과 '헬스/뷰티' 섹션의 가격들이 **전체적으로 정렬된 것은 아니다.** 
따라서 `price = 20000`인지 일일이 확인하는 추가작업을 해야함으로 인덱스의 성능을 절반만 활용하게 되는 셈이다.
