## 핵심원칙: 카디널리티
- 카디널리티가 높다: 해당 칼럼에 중복되는 값이 거의 없다는 의미
- 카디널리티가 낮다 : 해당 칼럼의 값이 몇 종류 안되어 중복되는 값이 많다.

## 인덱스 생성 가이드라인
1. `WHERE` 절에서 자주 사용되는 칼럼
	- 인덱스의 존재 이유 자체가 `WHERE`절의 검색 속도를 높이는 것이기 때문이다.
	- 사용자들이 상품을 검색할 때 `items.item_name`으로 검색하거나, 특정 카테고리를 필터링한다면 이 칼럼들은 인덱스 생성의 우선 후보가 된다.
2. `JOIN`의 연결고리가 되는 칼럼(외래 키)
3. `ORDER BY`절에서 자주 사용되는 칼럼
	- `ORDER BY`를 사용한 정렬은 데이터의 양이 많을 경우 매우 비용이 큰 작업이다. 
	- 데이터베이스는 결과를 반환하기 전에 모든 데이터를 메모리에 올리고 정렬해야 하기 때문이다.

## 인덱스의 단점과 주의사항
1. 저장 공간
	- 원본 테이블과 별개로 B-Tree구조를 가진 물리적인 파일로 디스크에 저장된다.
	- 원본 테이블 크기의 약 10% 내외의 공간을 추가로 차지한다고 알려져있다.
2. 쓰기 성능
	- 인덱스는 `SELECT`속도를 높이는 대가로, `INSERT`, `UPDATE`, `DELETE`의 속도를 희생시킨다.
	- `INSERT`
		- 해당 테이블에 생성된 모든 인덱스에도 B-Tree에 대한 키 값과 주소가 추가되어야한다. 만약 인덱스가 5개라면 테이블 삽입도 5번 작업이 추가되는 셈이다.
	- `DELETE`
		- 데이터가 삭제되면 모든 인덱스에서도 해당 칼럼에 대한 키 값이 삭제되어야한다.
	- `UPDATE`
		- 만약 인덱스가 있는 칼럼의 값이 변경된다면 기존 인덱스 항목을 삭제하고 새로운 칼럼의 값으로 인덱스 항목을 '추가'하는 것과 유사한 작업을 수행한다.
		- 즉, `INSERT`와 `DELETE`가 동시에 발생하는 것과 같다.
