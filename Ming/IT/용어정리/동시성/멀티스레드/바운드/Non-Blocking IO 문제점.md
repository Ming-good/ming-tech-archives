-  작업 스레드의 종료 여부와 관계없이 요청한 스레드는 차단되지 않는다
-  만약에 스레드 내부에 CPU를 많이 사용하는 작업이 포함된 경우 성능에 악영향을 준다.
	- Non-Blocking I/O 모델에서는 보통 **소수의 스레드**가 많은 연결을 처리.
	- 그런데 한 스레드에서 **CPU를 오래 점유하는 연산**이 발생하면,  그 스레드가 맡은 다른 I/O 작업 처리도 지연됨 → 병목 현상 발생.
	- 예: Netty 이벤트 루프에서 대규모 JSON 파싱, 복잡한 암호화 계산 등을 직접 처리하는 경우.
- 사용자의 요청에서 응답까지의 전체 과정에 Blocking I/O 요소가 포함된 경우 Non-Blockin의 이점을 발휘하기 힘들다.
