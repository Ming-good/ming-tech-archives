
---
**데이터**가 있으면 **추상 클래스**,  **역할**이 중요하면 **인터페이스**

> **인터페이스** : 능력(행동)을 정의
> **추상클래스** : 공통 상태(필드)와 행동을 함께 정의 가능
---

## 기능 비교

| 기능                 | 인터페이스 (Java 8~) | 추상 클래스       |
| ------------------ | --------------- | ------------ |
| 추상 메서드 정의          | ✅ 가능            | ✅ 가능         |
| default 메서드(기본 구현) | ✅ 가능 (Java 8~)  | ✅ 가능         |
| static 메서드 정의      | ✅ 가능            | ✅ 가능         |
| 필드(상태) 정의          | ❌ (상수만 가능)      | ✅ 인스턴스 변수 가능 |
| 생성자                | ❌ 없음            | ✅ 있음         |
| 다중 상속 가능성          | ✅ 다중 구현 가능      | ❌ 단일 상속만 가능  |

## 선택 기준 판단표 요약

|상황|선택|
|---|---|
|**공통 필드(상태)** 공유 필요|🔸 추상 클래스 (`abstract class`)|
|**공통 기본 구현** + 생성자|🔸 추상 클래스|
|**기능/능력 표현 (역할 중심 설계)**|🔸 인터페이스 (`interface`)|
|**다형성 극대화 (전략 패턴, DI, Mock)**|🔸 인터페이스|
|**다중 상속 필요**|🔸 인터페이스|

## 선택 기준 판단표
|판단 기준 항목|추상 클래스 (`abstract class`)|인터페이스 (`interface`)|
|---|---|---|
|✅ 공통 필드(상태) 필요함?|**O**: 인스턴스 변수 정의 가능|**X**: `public static final` 상수만 가능|
|✅ 생성자 필요함?|**O**: 생성자 정의 가능|**X**: 생성자 없음|
|✅ 공통 기본 로직이 많은가?|**O**: 필드 + 메서드로 공통 구현 제공 가능|**△**: `default` 메서드로 일부 구현은 가능하지만 제한적|
|✅ 다중 상속 필요함?|**X**: 클래스는 1개만 상속 가능|**O**: 인터페이스는 여러 개 구현 가능|
|✅ 역할/능력의 표현인가?|**△**: 표현은 되지만 권장되지 않음|**O**: "할 수 있다"의 능력 표현에 적합 (`Runnable`, `Serializable` 등)|
|✅ 계층 구조(상속 체계) 설계인가?|**O**: 계층적 상속 설계에 적합|**X**: 계층보다는 능력 조합 방식|
|✅ 후속 확장을 고려하고 있는가?|**O**: 기본 동작, 필드 제공이 중요할 때|**O**: 외부에 명확한 API 계약 제공할 때|
|✅ 테스트 시 Mock 만들 가능성?|**△**: 가능은 하지만 제약 있음|**O**: 인터페이스 기반이면 더 쉬움|
|✅ 라이브러리나 프레임워크용 API 설계?|**△**: 내부 기본 로직 필요할 땐 가능|**O**: 사용자에게 “이렇게만 구현하면 된다”는 계약 제공에 최적|

## 추상클래스 사용 예시
#### 1. 공통 상태(필드)가 필요할 때

```
public abstract class Animal {
    protected String name;

    public Animal(String name) {
        this.name = name;
    }

    public abstract void sound();
}
```

#### 2. 생성자 로직이 필요할 때
```
public abstract class Service {
    protected final Logger logger;

    public Service() {
        this.logger = LoggerFactory.getLogger(getClass());
    }
}
```
