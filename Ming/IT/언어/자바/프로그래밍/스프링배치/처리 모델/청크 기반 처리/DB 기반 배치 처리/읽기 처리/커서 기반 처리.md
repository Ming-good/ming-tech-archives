- 데이터베이스와 연결을 유지하면서 데이터를 순차적으로 가져온다
- 하나의 커넥션으로 데이터를 스트리밍하듯 처리한다
- 메모리는 최소한으로 사용하면서 최대한의 성능을 뽑아낸다
## `JdbcCursorItemReader`구조
![[Pasted image 20251016140041.png]]
![[Pasted image 20251016140249.png]]
- `rowMapper`: 결과 데이터를 객체로 전환
	- `BeanPropertyRowMapper`: 전통적인 setter기반 매핑 방식
	- `DataClassRowMapper`: Java Record같은 불변 객체를 위해 설계된 구현체이다
	- Custom `RowMapper`: 직접 구현 사용
- `PreparedStatement`: 데이터베이스에 쿼리를 실행하고 그 결과를 `ResultSet`으로 가져오는 JDBC 핵심 컴포넌트이다.

## 사용 예시
![[Pasted image 20251016140640.png]]
```java
// Setter 기반 
@Bean
public JdbcCursorItemReader<Victim> terminatedVictimReader() {
	return new JdbcCursorItemReaderBuilder<Victim>()
			.name("terminatedVictimReader")
			.dataSource(dataSource)
			.sql("SELECT * FROM victims WHERE status = ? AND terminated_at <= ?")
			.queryArguments(List.of("TERMINATED", LocalDateTime.now()))
			.beanRowMapper(Victim.class)
			.build();
}

// Custom 기반
@Bean
public JdbcCursorItemReader<Victim> terminatedVictimReader() {
    return new JdbcCursorItemReaderBuilder<Victim>()
            .name("terminatedVictimReader")
            .dataSource(dataSource)
            .sql("SELECT * FROM victims WHERE status = ? AND terminated_at <= ?")
            .queryArguments(List.of("TERMINATED", LocalDateTime.now()))
            // 람다로 RowMapper 구현
            .rowMapper((rs, rowNum) -> {
                Victim victim = new Victim();
                victim.setId(rs.getLong("id"));
                victim.setName(rs.getString("name"));
                victim.setProcessId(rs.getString("process_id"));
                victim.setTerminatedAt(rs.getTimestamp("terminated_at").toLocalDateTime());
                victim.setStatus(rs.getString("status"));
                return victim;
            })
            .build();
}
```
## 동작 상세
##### **`ResultSet`내부 버퍼링**
-  JDBC 드라이버는 기본적으로 여러 개의 row를 미리 가져와 ResultSet의 내부 버퍼에 저장해둔다.
- `ResultSet.next()`가 호출될 때마다 이 버퍼에서 데이터를 하나씩 꺼내는 구조다.
##### **`JdbcCurosrItemReader.read()`** 과정
- 버퍼 확인 후, 버퍼에 데이터가 있으면, 데이터베이스 통신 없이 즉시 반환
##### **스냅샷 읽기**
- `JdbcCursorItemReader`가 데이터가 중간에 수정되더라도 처음 조회한 데이터 그대로 읽게 된다.
##### **`JdbcCursorItemReader`의  `ORDER BY` 중요성**
- `Step`실패 시 `jumpToItem()`메서드 호출을 통해 실패 지점부터 다시 시작한다.
	- `jumpToItem()`: 내부적으로 `ResultSet.next()`를 호출해서 실패 지점 까지 커서를 이동 시킨다.
- `ORDER BY`로 정렬이 되지 않고 재 시작 시, `jumpToItem()`이 올바른 실패 지점 까지 도달하는 것을 보장할 수 없다.
