`skipLimit`횟수 까지 건너뛸 수 있으며 횟수를 초과했을 경우 오류를 발생시킨다.
```java
@Bean
public Step terminationRetryStep() {
    return new StepBuilder("terminationRetryStep", jobRepository)
            .<Scream, Scream>chunk(3, transactionManager)
            .reader(terminationRetryReader())
            .processor(terminationRetryProcessor())
            .writer(terminationRetryWriter())
            .faultTolerant()
            .skip(TerminationFailedException.class)
            .skipLimit(2) // 두 번까지만 참아주겠다 💀
            .build();
}
```

## `ItemReader`에서 건너뛰기 동작
- 재시도 기능이 없는 `ItemReader`에서는 `read()`메서드 호출 중 예외가 발생하면, 
	- 이 예외를 catch하고 SkipPolicy를 사용해 해당 예외가 건너뛰기 가능한지 판단한다.
	- 건너뛰기 가능하다고 판단되면 예외를 무시하고 바로 다음 아이템 읽기를 진행한다,
- `ItemReader`에서 아이템을 건너뛰더라도 `ItemProcessor`로 전달되는 청크 내 아이템의 개수는 변하지 않는다.
	- 예를들어 청크 사이즈가 3으로 설정되어있다면, 건너뛰기가 발생해도 "유효한 하이템 3개를 채울 때까지" `read()`메서드를 호출한다.

## `ItemProcessor`에서 건너뛰기 동작
- `ItemProcessor`에서 `null`을 반환하면 필터링 되어 `ItemWriter`에 전달되지 않는다
- 스프링 배치에서 필터링된 아이템은 `filterCount`에, 건너뛰기 된 아이템은 스킵 카운터(`skipCount`)에 기록한다.
> 건너뛰기 횟수는 스텝 전체에서 발생한 총 횟수로 관리되기 때문에 `skipLimit`를 너무 낮게 설정하면 작업이 일찍 끝나서 실패할 수 있다.

## `ItemWriter`에서 건너뛰기 동작
- `ItemProcessor`에서는 개별 아이템을 처리함으로 문제 지점을 바로 식별하고 건너뛸 수 있었지만, `ItemWriter`에서는 청크단위로 수행함으로, 예외 발생시 어떤 아이템이 문제인지 즉시 알 수 없다.
- 따라서 `ItemWriter`에서는 **재시도가 불가능하다고 판단되면 스캔모드로 돌입**한다.
#### 스캔모드란?
내 아이템을 하나씩 개별 처리하는 특수 모드다.
![[Pasted image 20251104233000.png]]
#### 스캔 모드에서 Skip 프로세스
1. 예외 발생 및 재시도 판단
	- `ItemWriter`에서 예외가 발생하면 스프링 배치는 재시도 가능 여부를 확인한다,.
2. 스캔 모드 전환
	- 재시도 불가능하다고 판단되면 스캔 모드로 전환되고 청크 트랜잭션이 롤백된다.
3. 개별 아이템 처리
	- 스캔 모드에서 청크 내 모든 아이템을 하나씩 개별처리한다. 
	- `ItemProcessor` -> `ItemWriter` -> 즉시 커밋을 각 아이템마다 반복한다.
4. 문제 아이템 건너뛰기
	- 개별 처리 중 예외가 발생한 아이템은 SkipPolicy판단을 거쳐 건너뛴다.
5. 스캔 모드 종료
	- 청크 내 모든 아이템에 대한 개별 처리가 완료되면 스캔 모드가 종료된다.
#### 스캔 모드 성능 비용
> 스캔 모드는 강력한 무기지만 그만큼 성능상 댓가가 크다. 청크 사이즈가 1000이었다면 스캔 모드에서 무료 1000번의 개별 커밋이 발생한다.

## `SkipListener`
```java
public interface SkipListener<T, S> extends StepListener {

    default void onSkipInRead(Throwable t) {
    }

    default void onSkipInWrite(S item, Throwable t) {
    }

    default void onSkipInProcess(T item, Throwable t) {
    }
}
```
#### 제네릭 타입
- `T` : `ItemProcessor`에 전달되는 아이템의 타입이다.
- `S` : `ItemWriter`에 전달되는 아이템의 타입이다.
