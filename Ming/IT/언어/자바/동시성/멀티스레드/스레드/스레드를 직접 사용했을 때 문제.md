### 1. 스레드 생성 비용으로 인한 성능 문제
- **메모리 할당** : 각 스레드는 자신만의 호출 스택을 가지고 있어야 한다. 이 호출 스택은 스레드가 실행되는 동안 사용하는 메모리 공간이다. 따라서 스레드를 생성할 때 이 호출 스택을 위한 메모리를 할당해야한다.
- **운영체제 자원 사용** : 스레드를 생성하는 작업은 운영체제 커널 수준에서 이루어지며, 시스템 콜을 통해 처리된다. 이는 CPU와 메모리 리소스를 소모하는 작업이다.
- **운영체제 스케줄러 설정** : 새로운 스레드가 생성되면 운영체제의 스케줄러는 이 스레드를 관리하고 실행 순서를 정해야 한다. 이는 운영체제의 스케줄링 알고리즘에 따라 추가적인 오버헤드가 발생할 수 있다.
>- 스레드 하나는 보통 1MB 이상의 메모리를 사용한다.

##### 요약
- 어떤 작업 하나를 수행할 때 마다 스레드 각각 생성하고 실행한다면, 스레드 생성비용 떄문에 시간이 소모된다. 가벼운 작업의 경우 작업 실행 시간보다 스레드 생성시간이 더 오래 걸릴 수 있다.
  
### 2. 스레드 관리 문제
- CPU와 메모리가 버틸 수 있는 최대 스레드의 수 까지만 스레드를 생성하고 관리해야한다. (시스템 자원 이상 스레드 생성시 시스템이 버티지 못 한다.)
- 안전한 종료 처리(실행중이 스레드가 남은 모든 작업을 수행후 프로그램을 종료하고 싶다던가 급한 종료시 인터럽트를 사용해야 한다던가)가 필요하다면 관리 되어야한다.

### 3.Runnable 인터페이스의 불편함
- **반환 값이 없다** : run() 메서드는 반환 값을 가지지 않는다. 따라서 실행 결과를 얻기 위해서는 별도의 메커니즘을 사용해야 한다.(스레드 실행 결과를 멤버 변수에 넣고 join()등을 통해 스레드 종료후 결과값을 받아야 한다.)
- **에외 처리** : run() 메서드는 체크 예외를 던질 수 없다. 체크 예외 처리는 내부에서 처리해야한다.

>1. 작업 스레드는 간단히 값을 `return` 을 통해 반환하고, 요청 스레드는 그 반환 값을 바로 받을 수 있다면 코드가 훨씬 더 간결해질 것이다.
>2. 이런 문제를 해결하기 위해 Executor 프레임워크는 [[Callable]] 과 [[Future]] 라는 인터페이스를 도입했다.
